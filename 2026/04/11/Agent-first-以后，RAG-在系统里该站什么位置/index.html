Agent-first 以后，RAG 在系统里该站什么位置

2026-04-11 5 分钟阅读 1838 字

传统的“一次性召回并回答”在处理复杂文档依赖时存在局限性。在 Knowledge Hub 中，我们将 RAG 重构为由 Agent 调度的“证据源”，通过 Scout（侦察）与 Deep Read（精读）的分层协议，彻底解决了检索片段的上下文断裂难题。

AI knowledge-hub Agent RAG MCP

在 RAG（检索增强生成）系统的演进过程中，很多开发者会发现：无论如何调优 Embedding 模型或切片策略，召回率的提升终究会遇到天花板。这往往是因为单一的检索逻辑难以承载复杂的、跨文档的业务推理。

在 Knowledge Hub (KH) 的实践中，我们意识到：RAG 模块不应是直接生成答案的“大脑”，而应是支持 Agent 取证的“证据库”。 这种理念的转变，促使我们将整个系统架构推向了 Agent-first 模式。

一、标准化的武器库：基于 MCP 的工具链

在 KH 的 internal/agent 模块中，我们并没有随意编写 Agent 调用的函数，而是严格遵循了 MCP (Model Context Protocol) 这一行业标准。

在 pkg/mcp 中，我们将所有的检索与内容获取能力封装为标准化的 Tool。这种设计让 Agent 具备了统一的交互界面：

  • search_knowledge_base：证据搜索与定位。
  • read_document：全文内容消费与理解。
  • get_folder_info：获取 VFS 目录树上下文。

通过 MCP 协议，RAG 从一个封闭的查询接口，变成了一个可被 Agent 自由组合、按需调用的“标准武器库”。

二、双阶段取证协议：Scout（侦察）与 Deep Read（精读）

在 KH 的 internal/agent/service.go 中，我们通过 toolUsagePrompt 固化了一套名为 “渐进式披露 (Progressive Disclosure)” 的工作流。这套工作流将检索过程拆解为三个优先级极高的阶段：

2.1 阶段零：记忆扫描 (Memory Scan)

Agent 被强制要求先核对对话历史（History）。如果历史上下文中已存在答案，严禁再次调用检索工具。这极大地降低了 Token 消耗并提升了响应速度。

2.2 阶段一：Scout（广泛侦察）

当历史记忆不足时，Agent 启动侦察阶段。此阶段调用 search_knowledge_base。它不追求直接获取结论，而是通过语义搜索“侦察”出可能包含答案的 doc_id 及其关键片段。
强制规则：如果第一次搜索结果为空，Agent 被严禁直接回复“找不到”，必须至少重试 2 次（改写 Query 或精简关键词）后才能放弃。

2.3 阶段二：Deep Read（定向精读）

这是解决 RAG 片段化问题的核心。我们告诉 Agent：检索返回的只是片段，不是完整事实。
如果片段不足以支撑结论，Agent 必须从片段头部提取 doc_id，并主动调用 read_document 读取全文。在充分理解全文上下文后，再给出最终回答。这种“两段式”取证，彻底消除了传统 RAG 因片段截断导致的“脑补”和“幻觉”现象。

三、Strict Grounding：坚不可摧的“硬约束”

在企业级问答中，AI 的“聪明”有时是致命的危险。我们通过 strictGroundingPrompt 为 Agent 戴上了沉重的枷锁：

  1. 事实归正：回答必须严格基于知识库内容，不得使用任何外部常识。
  2. 证据优先：没有检索到支撑证据时，必须诚实回复“知识库中未找到相关信息”，严禁编造流程、时间或政策。
  3. 可追溯性：每一个回答都必须能映射回 pkg/mcp 工具链返回的原始路径。

这种硬约束将 RAG 的功能定位从“生成器”降级为“校验器”，但却极大地提升了系统的业务鲁棒性。

四、Agentic Traversal：控制权与确定性的博弈

为了防止 Agent 在复杂的目录树中“迷路”或无限循环，我们在 dataset.go 的 AgenticConfig 中引入了关键参数：

  • max_steps & max_depth：限制 Agent 取证的深度和步数，防止死循环。
  • max_context_tokens：动态管理 Agent 的“阅读容量”，在精读与性能之间寻找平衡。

在 adapter.go 的底层实现中，我们将中间的取证过程（即工具调用的 JSON 报文）进行了“静默处理”。用户在前台看到的是流畅的、经过深度思考后的结论，而系统内部则经历了一场严密的、基于 MCP 协议的取证长征。

总结

Agent-first 架构的本质是：承认检索是不完美的，并用大模型的推理能力来补偿这种不完美。

通过将 RAG 模块重构为受控的、标准化的证据源，并引入 Scout/Deep Read 协议与 Strict Grounding 约束，KH 成功地将 RAG 从一个单纯的技术点，升级为了一个具备自愈能力和工程确定性的知识治理体系。在这个体系下，RAG 依然重要，但它已不再是孤军作战，而是 Agent 指挥下的精锐尖兵。
