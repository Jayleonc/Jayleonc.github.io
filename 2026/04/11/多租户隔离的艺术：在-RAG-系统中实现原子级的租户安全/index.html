多租户隔离的艺术：在 RAG 系统中实现原子级的租户安全

2026-04-11 5 分钟阅读 1946 字

数据安全是企业级系统的底线。本文深入解析 Knowledge Hub 如何通过 GORM Scope、表名自动推断与 1=0 硬熔断机制，实现多租户数据的物理隔离。

AI knowledge-hub 多租户 安全隔离

在做企业级 RAG 系统时，多租户隔离 (Multi-tenancy Isolation) 是一道永远绕不过去的“鬼门关”。

我有一次在做架构审计时曾问过团队一个非常刻薄的问题：“如果你在写检索 JOIN 语句时，漏写了一个 tenant_id 过滤，A 公司的机密会不会在 B 公司的 AI 对话框里蹦出来？” 整个会议室瞬间陷入了死寂。这种由于人为疏忽导致的“数据越权”，在 RAG 这种高度依赖语义模糊匹配的系统里，几乎是毁灭性的。

为了守住这条底线，Knowledge Hub 的核心设计哲学是：多租户隔离绝对不能靠开发者的自觉，必须下沉到数据库的最底层，变成一种自动化的、原子级的物理隔离。

架构的洁癖：为什么应用层过滤是“皇帝的新衣”？

很多团队习惯于在应用层（Service 层）写各种逻辑判定：if user.TenantID != doc.TenantID { return err }。或者是依赖开发者的习惯，在每个 SQL 后面手动挂一个 .Where("tenant_id = ?", tid)。

但在 Knowledge Hub 的架构里，我把这种模式视为“极高风险”。随着业务逻辑日益复杂，尤其是涉及到 VFS 2.0 的多表联查、向量数据库的 Raw SQL 检索时，人为遗漏几乎是必然发生的。

我们的方案是利用 GORM 的 Scope 机制，在 internal/database/scope/acl.go 中构建了一套“查询注入引擎”。

自动推断：解决 SQL 别名的“歧义陷阱”

在复杂的检索 SQL 中，我们经常会 JOIN 三四个表。如果你简单地全局注入一个 tenant_id 过滤，数据库会立即报错：column "tenant_id" is ambiguous。

在 applyACLScope 的实现中，我们引入了一套表名自动推断逻辑。
系统会动态捕获 db.Statement.Table。如果是多表联查，它能根据当前的 Statement 自动为 tenant_id 补上前缀（比如 "document".tenant_id）。

这种细碎到极致的 SQL 构造能力，确保了多租户校验就像是水和空气一样，无孔不入地渗透进了系统的每一条数据库链接。开发者在写业务逻辑时，根本不需要（也不被允许）去操心租户过滤，系统会在执行前自动帮你把这道防线筑牢。

Fail-Safe：那个让我晚上睡得踏实的“1 = 0”

在安全性设计上，我一直推崇 Fail-Safe（安全失败） 原则。

为什么要写这种看起来很笨的 1 = 0？
因为在分布式系统里，上下文丢失（Context Loss）是一个非常隐蔽的风险。如果因为某种原因（比如中间件拦截失效、Token 异常），查询进入了底层但失去了租户 ID，传统的逻辑可能会因为没有过滤条件而导致“全量扫描”，从而造成巨大的泄露。

而 1 = 0 相当于在数据库层拉了一道死闸。只要你的身份标识（PrincipalID）或租户信息有任何瑕疵，这个查询在数据库端执行的结果集永远是零。这种“宁可杀错、绝不放过”的硬核熔断，才是我心里企业级系统该有的底气。

向量检索的“物理隔离”与“联防联动”

向量数据库（如 pgvector）由于其查询的特殊性（常涉及特定插件语法），往往是安全隔离的盲区。我们采取了“双重保险”策略：

  1. 参数层强检：在进入检索函数前，首先调用 module.GetSpace(ctx)。如果获取不到有效的租户身份，请求在应用层就会被熔断。
  2. 实时 JOIN 过滤：这是最关键的一环。在执行向量相似度计算的 SQL 语句中，我们强制执行一个 JOIN document 表。

由于 document 表已经挂载了上述的 GORM Scope，这本质上形成了一次“联防联动”：哪怕向量切片本身由于各种原因出现隔离缝隙，只要其所属的父级文档（Document）在当前租户下不可见，整个结果集就会被物理屏蔽。

结语：看不见的安全屏障

在 Knowledge Hub 中，多租户隔离不再是一个需要开发者在每一行代码里去反复横跳的功能点，而是一项“基础设施”。

我们通过 GORM Scope 实现了对业务代码的零侵入安全保障。这种看不见的安全屏障，不仅减少了代码的冗余，更重要的是，它将整个系统的安全边界从“人的习惯”转移到了“程序的确定性”上。在处理那些极其敏感的企业私域知识时，只有这种原子的、物理的、不讲情面的隔离，才能真正赢得用户的信任。
